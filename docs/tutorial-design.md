# 前端工程师的 AI Agent 实战指南：TypeScript 版（教程设计文档）

## 1. 项目定位

本教程面向已经具备基础 TypeScript 开发经验、但尚未系统构建过 AI Agent 的开发者，目标是带读者从零实现一个可运行、可扩展、可调试的 Agent，并在这个过程中逐步建立对 Agent 核心能力的正确心智模型。

教程的核心定位不是“罗列概念”，而是“沿着能力增长路径搭建 Agent”。读者完成整套教程后，不仅能解释 ReAct、Plan-and-Execute、RAG、MCP、Skill、上下文工程这些概念，还应该能够在 TypeScript 工程中把它们真正串起来。

一句话定位：

> 用 TypeScript 从零实现一个现代 AI Agent，并理解它为什么这样设计。

## 2. 目标读者

### 2.1 读者画像

- 能读写 TypeScript，理解异步、模块、接口、泛型等基础语法
- 用过 LLM API 或至少理解 Chat Completion / Responses API 的基本工作方式
- 还没有系统构建过 Agent，或者只写过最简单的聊天机器人
- 希望理解 Agent 的工程化实现，而不是停留在产品层面或概念层面

### 2.2 先验要求

教程默认读者已经知道：

- 什么是大语言模型
- Prompt 是什么
- JSON Schema / Zod 这类结构化约束的基本用途
- Node.js/TypeScript 项目如何运行

教程不要求读者预先掌握：

- Agent 框架
- 向量数据库
- MCP 生态
- 多 Agent 协作

## 3. 教学目标与边界

### 3.1 教学目标

读者完成教程后，应能掌握以下能力：

1. 理解 Agent 与普通聊天机器人、固定工作流之间的边界
2. 能独立实现最小可运行的 Agent Loop
3. 能为 Agent 设计稳定的工具调用接口
4. 能理解并实现 ReAct 与 Plan-and-Execute 两种执行模式
5. 能设计合理的上下文拼装和压缩策略
6. 能为 Agent 接入外部知识，并理解 RAG 在其中的角色
7. 能理解 MCP 的价值，并在 TypeScript 中接入 MCP Server
8. 能把重复任务沉淀成 Skill，而不是把所有逻辑堆进一个 Prompt
9. 能把这些能力整合成一个可用、可调试、可扩展的 Agent Demo

### 3.2 非目标

为了控制教程节奏，本教程不追求：

- 覆盖所有 Agent 框架的 API 用法
- 深入所有向量检索算法或索引优化策略
- 构建企业级生产系统
- 讨论所有前沿论文或学术术语
- 展示“无所不能”的大而全 Agent

本教程更关注“第一性原理 + 可运行实现 + 工程权衡”。

## 4. 设计原则

### 4.1 按能力增长路径组织，而不是按术语分类

教程不按照“先讲所有概念，再做代码”的方式展开，而是让读者随着 Agent 能力逐步增长，依次遇到并解决问题：

- 先让 Agent 跑起来
- 再让 Agent 学会行动
- 再让 Agent 学会规划
- 再让 Agent 学会看见正确的信息
- 再让 Agent 接入外部能力与知识
- 最后让 Agent 的能力可以复用、扩展与落地

### 4.2 每章都要有“最小闭环”

每一章都必须包含一个可运行的最小闭环，而不是只有概念解释。读者应该在每章结束时都获得一个明确成果，例如：

- 一个最小 Agent Loop
- 一个可校验的工具注册器
- 一个支持规划的执行器
- 一个本地知识检索模块
- 一个接入 MCP 的 Agent

### 4.3 深入浅出，不堆术语

教程中的每个高级概念都要先用工程问题来引出。例如：

- 不先讲 Context Engineering 的定义，而是先讲“为什么历史消息越多，Agent 越容易做错”
- 不先讲 MCP 协议字段，而是先讲“为什么工具接入一多，项目会变得不可维护”
- 不先讲 Skill 的抽象意义，而是先讲“为什么不能把所有任务都塞进系统提示词”

### 4.4 强调边界与权衡

教程必须持续提醒读者：

- ReAct 不是万能方案
- Plan-and-Execute 不是“更高级”的 ReAct
- RAG 不是记忆系统的替代品
- MCP 不是工具调用本身，而是标准化接入层
- Skill 不是一句更长的 Prompt

### 4.5 代码实现优先于框架依赖

教程优先自己实现关键骨架，避免一开始就依赖大型 Agent 框架。原因有两个：

- 读者能真正理解 Agent 的工作原理
- 后续迁移到任意框架时，知道哪些是核心抽象，哪些只是框架包装

## 5. 贯穿案例设计

整套教程建议围绕一个统一案例展开，避免每章换一个全新主题导致认知断裂。

推荐案例：

> 构建一个“TypeScript 研究与工程助手”，它能够理解任务、调用工具、读取知识库、接入外部资源，并完成多步问题求解。

这个案例适合作为主线，因为它天然覆盖以下能力：

- 需要 Agent Loop
- 需要工具调用
- 需要多步规划
- 需要读取项目文档或知识库
- 适合演示 RAG
- 适合接入 MCP Server
- 适合把“代码审查”“资料调研”“文档总结”封装成 Skill

### 5.1 贯穿案例中的任务示例

在不同章节中，可以用同一 Agent 完成不同层级的任务：

- 第 1 阶段：回答简单任务，必要时调用单个工具
- 第 2 阶段：查询文件、读取文档、执行简单命令
- 第 3 阶段：把复杂任务拆成计划并逐步执行
- 第 4 阶段：根据本地知识库回答工程问题
- 第 5 阶段：通过 MCP 接入标准化资源与工具
- 第 6 阶段：通过 Skill 复用“代码审查”“技术调研”等任务模板

### 5.2 主线案例的教学价值

这个案例的价值在于，读者不会感觉自己在学一堆分散的能力，而是在不断升级同一个 Agent。

## Part I. 建立正确心智模型

### 第 01 章：为什么需要 Agent，而不只是 Chat API

#### 本章目标

帮助读者建立边界感，明确什么问题值得用 Agent，什么问题不值得。

#### 关键问题

- 聊天机器人、工作流、Agent 有什么区别
- 为什么“能对话”不等于“能行动”
- Agent 解决的是哪类问题

#### 关键概念

- LLM
- Chatbot
- Workflow
- Agent
- State
- Action
- Observation

#### 读者产出

- 能判断一个任务是否需要 Agent
- 能解释 Agent 的最小构成：状态、模型、动作、观察、停止条件

#### 建议内容结构

1. 从“问答系统”讲起
2. 为什么固定工作流会失效
3. Agent 的最小定义
4. 一个简单任务在 Chatbot / Workflow / Agent 中的差别
5. 本教程主线案例介绍

#### 本章避免事项

- 不要一开始就塞入大量术语
- 不要把 Agent 描述成“更智能的聊天机器人”

### 第 02 章：从零写一个最小 Agent Loop

#### 本章目标

让读者理解 Agent 的核心不是框架，而是一个清晰的循环。

#### 关键问题

- Agent 为什么本质上是一个循环
- 消息历史、工具请求、工具结果如何串起来
- 什么情况下结束当前任务

#### 关键概念

- Agentic Loop
- Message History
- Stop Reason
- Tool Call
- Tool Result
- State Transition

#### 代码产出

- 一个最小的 `runAgent()` 函数
- 一个支持“模型请求工具 -> 执行工具 -> 回填结果”的循环
- 基础日志输出

#### 示例实现建议

- 只接入 1 个工具，比如 `searchDocs` 或 `getWeather`
- 先不追求通用性，先跑通闭环

#### 读者产出

- 真正运行起第一个 Agent
- 知道最小闭环中每一轮到底发生了什么

#### 本章避免事项

- 不要在这里提前加入计划器、记忆系统、RAG

## Part II. 让 Agent 学会行动与规划

### 第 03 章：Tool Calling，给 Agent 行动力

#### 本章目标

帮助读者设计稳定的工具接口，而不是“把任意函数暴露给模型”。

#### 关键问题

- 工具为什么需要结构化参数
- 为什么工具返回值必须服务下一轮推理
- 工具失败后怎么让 Agent 可恢复

#### 关键概念

- Tool Definition
- Input Schema
- Output Contract
- Validation
- Timeout
- Retry
- Idempotency

#### 代码产出

- 一个 `tool registry`
- 基于 TypeScript 类型和 Zod 的参数校验
- 标准化工具返回结构
- 工具执行错误的包装层

#### 建议示例工具

- `searchFiles`
- `readFile`
- `fetchUrl`
- `writeTodo`

#### 本章重点

重点不是“接很多工具”，而是让读者理解怎样设计一个适合模型使用的工具接口。

---

### 第 04 章：ReAct，让 Agent 学会边想边做

#### 本章目标

让读者理解 ReAct 适合什么任务，以及它为什么在开放式任务中有效。

#### 关键问题

- 为什么单轮决策不足以解决复杂任务
- ReAct 如何通过“行动 -> 观察 -> 再决策”逐步逼近答案
- 什么任务适合 ReAct

#### 关键概念

- Reason + Act
- Iterative Execution
- Observation-Driven Decision
- Short Horizon Tasks

#### 代码产出

- 基于上一章的工具系统实现 ReAct 执行器
- 支持多轮工具调用
- 增加基础轮次限制和终止保护

#### 示例任务

- “阅读项目里的配置文件，告诉我这个项目怎么启动”
- “先找文档，再总结重点”

#### 本章重点

ReAct 要讲成“基于观察逐步前进的执行模式”，而不是“展示模型的思维过程”。

#### 本章避免事项

- 不展示冗长的思维链文本
- 不把 ReAct 包装成最终答案

---

### 第 05 章：Plan-and-Execute，从反应式到规划式

#### 本章目标

帮助读者理解何时需要显式规划，以及规划器和执行器如何协作。

#### 关键问题

- ReAct 在长任务上的问题是什么
- 为什么有些任务需要先规划再执行
- 计划一定要固定不变吗

#### 关键概念

- Planning
- Executor
- Replanner
- Task Decomposition
- Long Horizon Tasks

#### 代码产出

- 一个 `planner`
- 一个 `executor`
- 一个 `replanner` 或最小重规划逻辑
- 计划结构体与执行状态结构体

#### 示例任务

- “帮我调研某个技术方案，并给出一页总结”
- “先列出要查看的材料，再逐项读取和整理”

#### 本章重点

要把 ReAct 与 Plan-and-Execute 放在同一张图里对比：

- ReAct：边走边看，适合开放式、短链路任务
- Plan-and-Execute：先拆解后执行，适合长链路、结构更清晰的任务

#### 本章避免事项

- 不把 Plan-and-Execute 描述成“更聪明的 Agent”
- 要明确说明它增加了复杂度、状态管理和失败面

## Part III. 让 Agent 看见正确的信息

### 第 06 章：Context Engineering，决定 Agent 的上限

#### 本章目标

让读者理解，很多 Agent 问题本质上不是模型不够强，而是上下文设计出了问题。

#### 关键问题

- 为什么上下文不是“塞得越多越好”
- 哪些信息应该进上下文，哪些不应该
- 如何管理工作记忆、历史消息和任务状态

#### 关键概念

- System Prompt
- Task Context
- Working Memory
- Conversation History
- State Injection
- Context Window
- Compression

#### 代码产出

- 一个 `buildContext()` 模块
- 分层组织上下文的策略
- 历史裁剪和摘要压缩逻辑
- 将工具结果、计划状态、用户目标注入上下文的机制

#### 示例内容

- 对比“原始历史全塞入”和“结构化上下文拼装”的效果
- 展示上下文错误如何导致工具误用

#### 本章重点

这是整套教程的中枢章节。要帮助读者建立一个稳定观念：

> Agent 的能力不仅取决于模型，还取决于它在每一轮看到了什么。

#### 本章避免事项

- 不把 Context Engineering 写成 Prompt Engineering 的同义词
- 不只谈提示词措辞，要谈状态组织与信息选择

---

### 第 07 章：RAG，给 Agent 补充外部知识

#### 本章目标

让读者理解 RAG 在 Agent 中的位置：它是上下文供给系统的一部分，而不是一个独立魔法模块。

#### 关键问题

- 为什么模型不能直接“知道”你的私有知识
- RAG 如何把外部文档变成可注入上下文的信息
- 检索结果为什么常常不稳定

#### 关键概念

- Chunking
- Embedding
- Retrieval
- Top-K
- Reranking
- Citation

#### 代码产出

- 文档切块脚本
- 向量化与索引构建流程
- 一个 `retrieve()` 模块
- 将检索结果注入 Agent 上下文的逻辑

#### 示例任务

- “根据内部文档回答这个问题，并给出引用来源”
- “查阅教程资料后，解释某个概念”

#### 本章重点

必须强调：

- RAG 的目标不是“增强记忆”
- RAG 的目标是在正确时机提供高相关上下文

#### 本章避免事项

- 不要把 RAG 只写成“向量数据库介绍”
- 不要跳过 chunk 设计和结果回填这两个真正影响效果的环节

## Part IV. 把能力变成工程系统

### 第 08 章：MCP，用标准协议接入工具与资源

#### 本章目标

帮助读者理解为什么当 Agent 工具和资源变多时，需要标准化接入层。

#### 关键问题

- MCP 想解决的核心问题是什么
- Tool、Resource、Prompt 在 MCP 中分别扮演什么角色
- 为什么 MCP 会让 Agent 更容易扩展和集成

#### 关键概念

- Model Context Protocol
- MCP Client
- MCP Server
- Tools
- Resources
- Prompts

#### 代码产出

- 一个最小 MCP Client
- 接入 1 到 2 个 MCP Server
- 把 MCP 暴露的工具和资源接入当前 Agent

#### 示例任务

- 从 MCP Resource 读取资料
- 调用 MCP Tool 获取外部结果
- 在 Agent 中统一路由本地工具和 MCP 工具

#### 本章重点

要让读者看到 MCP 的位置：

- Tool Calling 讲的是“Agent 如何调用工具”
- MCP 讲的是“工具和资源如何以标准方式接入 Agent”

#### 本章避免事项

- 不陷入某个特定平台的产品细节
- 不只解释协议概念，而要有最小接入示例

#### 示例 Skill

#### 本章重点

要让读者理解：

- Tool 是动作接口
- Skill 是任务能力封装

## 7. 推荐章节顺序与依赖关系

建议顺序如下：

1. 为什么需要 Agent
2. 最小 Agent Loop
3. Tool Calling
4. ReAct
5. Plan-and-Execute
6. Context Engineering
7. RAG
8. MCP
9. Skill
10. 整体组装

### 7.1 这样排序的原因

- `Agent Loop` 是全局骨架，必须最先建立
- `Tool Calling` 是行动基础，先于执行策略
- `ReAct` 是最自然的多轮执行模式，先讲更容易建立直觉
- `Plan-and-Execute` 建立在工具和循环之上
- `Context Engineering` 是中枢能力，放在执行模式之后最合适
- `RAG` 是上下文供给机制的一部分，因此应放在 Context Engineering 之后
- `MCP` 是工具与资源的标准化接入方式，应放在读者理解工具和上下文之后
- `Skill` 适合最后讲，因为它本质上是在前面能力之上的复用与沉淀

## 8. 每章统一写作模板

为了保证教程整体节奏一致，每章建议固定采用以下结构：

1. 这一章要解决什么问题
2. 如果没有这个能力，会出现什么工程痛点
3. 最小概念模型
4. 最小可运行实现
5. 逐步增强版本
6. 常见误区与边界
7. 本章总结
8. 下一章会解决什么问题

这个模板的好处是，读者能稳定形成预期，不会在章节之间不断切换阅读模式。

## 10. 是否要在教程主线中使用 LangChain、Vercel AI SDK 等框架

### 10.1 结论

本教程主线建议不直接依赖 LangChain、LangGraph、Vercel AI SDK 这类 Agent 开发框架来实现核心逻辑，而是优先用原生 TypeScript 自己搭建最小 Agent 骨架。

更准确地说：

- 教程主线不应把 Agent 框架作为理解 Agent 的前提
- 教程可以在进阶章节中介绍如何迁移到这些框架
- 如果只需要一层轻量模型调用封装，可以引入较薄的 SDK，但不应把循环、规划、上下文管理等核心能力完全交给框架隐藏

### 10.2 为什么主线教程不直接使用 Agent 框架

#### 原因 1：本教程的目标是建立第一性原理，而不是学习某个框架 API

这套教程的核心目标是帮助读者理解：

- Agent 为什么本质上是一个循环
- Tool 为什么需要清晰契约
- ReAct 和 Plan-and-Execute 为什么适用于不同问题
- Context Engineering 为什么决定 Agent 的上限

如果一开始就把这些能力全部包进框架里，读者往往会学会“怎么调用框架”，但不真正理解 Agent 的关键抽象。

#### 原因 2：框架会掩盖关键实现细节

许多 Agent 框架会自动处理：

- 消息拼装
- 工具调度
- 状态管理
- 多轮循环
- 中间结果回填

这些封装在做产品时很方便，但在教学中反而会遮住最重要的部分。对于入门读者来说，最容易形成误解的恰恰是这些被自动化的部分。

#### 原因 3：框架变化快，而核心原理更稳定

Agent 相关框架迭代速度很快，API 形态和推荐用法可能会持续变化。如果教程主线过度绑定某个框架，内容会更快过时。

相比之下，以下内容是更稳定的：

- Agent Loop
- Tool Contract
- Planner / Executor 分工
- Context 构建与压缩
- 检索结果如何回填到上下文

教程主线应该优先围绕这些稳定抽象展开。

#### 原因 4：读者后续迁移能力更强

如果读者先理解了骨架，再去看框架，会更容易识别：

- 哪些是 Agent 的共性能力
- 哪些只是特定框架的 API 包装
- 哪些地方应该依赖框架
- 哪些地方应该自己保留控制权

这会显著提升读者后续迁移到不同技术栈时的适应能力。

### 10.3 这并不意味着完全不用任何 SDK

不直接依赖 Agent 框架，不等于什么库都不用。

教程仍然可以合理使用：

- 模型调用 SDK
- Embedding SDK
- 向量检索库
- Zod 等类型与校验工具
- MCP Client SDK

这里的原则是：

> 可以依赖“基础设施库”，但不要在主线教学中把“Agent 的核心抽象”完全外包给框架。

### 10.4 针对不同框架的建议

#### Vercel AI SDK

Vercel AI SDK 非常适合 TypeScript/Next.js 生态，尤其在以下场景中很有价值：

- 流式输出
- Tool Calling
- Web UI 集成
- 前后端统一的 TypeScript 开发体验

如果本项目后续要从“教程示例”走向“可交互产品 Demo”，Vercel AI SDK 是值得优先考虑的。

但对于教程主线来说，它更适合作为：

- 模型调用层的工程增强方案
- 进阶章节中的迁移实践
- “如何把自研 Agent 接入现代 TS AI 工程栈”的补充章节

而不适合作为一开始就遮住底层结构的主实现方式。

#### LangChain / LangGraph

LangChain 和 LangGraph 更适合复杂编排场景，尤其是当系统开始出现以下需求时：

- 长流程任务
- 持久化状态
- 人工介入节点
- 多 Agent 协作
- 图状执行流
- 更复杂的运行时编排

它们很适合进阶项目或生产系统，但对于入门教程有两个明显问题：

- 抽象层更厚，学习成本更高
- 读者容易把“框架概念”误认为“Agent 原理”

因此，本教程不建议在主线中使用 LangChain / LangGraph 作为第一实现路径。

### 10.5 推荐策略

本教程建议采用如下策略：

#### 教程主线

- 使用原生 TypeScript 自己实现最小 Agent Runtime
- 自己实现 Loop、Tool Registry、Planner、Context Builder、Skill Loader 等核心部分
- 让读者直接看到 Agent 的基本骨架

#### 工程增强层

- 视需要引入模型调用 SDK 或较薄的基础设施封装
- 保持 Agent 核心控制逻辑仍然可见、可解释、可替换

#### 进阶章节

- 增加“如何把本教程中的 Agent 接入 Vercel AI SDK”
- 增加“什么时候该迁移到 LangGraph 或其他编排框架”

### 10.6 一句话判断标准

如果目标是教学，优先自己实现核心骨架。  
如果目标是快速构建 TypeScript 产品原型，可以优先考虑 Vercel AI SDK。  
如果目标是复杂编排、长流程状态和多角色运行时，可以再考虑 LangChain / LangGraph。

## 12. 关键教学难点与处理建议

### 难点 1：ReAct 和 Plan-and-Execute 容易被讲混

处理建议：

- 必须放在相邻章节
- 必须使用同一类任务做对比
- 必须讲清楚各自的适用边界

### 难点 2：RAG 容易被写成独立专题，和 Agent 主线脱节

处理建议：

- 明确把 RAG 放在 Context Engineering 之后
- 明确说明它是“上下文供给机制”

### 难点 3：MCP 容易变成协议术语堆砌

处理建议：

- 先讲工程问题，再讲协议
- 一定要给出最小接入示例

### 难点 4：Skill 容易沦为 Prompt 模板

处理建议：

- 通过反例说明“把所有任务都塞进系统提示词”的缺点
- 展示 Skill 如何复用工具、上下文策略、任务流程

### 难点 5：上下文工程容易被误解成“写提示词技巧”

处理建议：

- 重点讲信息选择、状态组织、压缩策略
- 减少“提示词措辞优化”比例

## 13. 内容风格建议

整套教程的语言风格建议保持以下特征：

- 解释清楚，但不过度学院化
- 少用大词，多用工程问题引导
- 尽量先给错误示例，再给改进方案
- 每章都保留一个“常见误区”小节
- 每章都强调“为什么这样设计”

推荐写法：

- 先讲问题
- 再讲最小模型
- 再讲代码
- 最后讲权衡

不推荐写法：

- 一开始堆定义
- 直接贴长代码不解释抽象边界
- 把“某框架的用法”写成“Agent 原理”

## 14. 推荐的首批落地计划

为了让项目从“设计阶段”顺利进入“写作阶段”，建议按以下顺序推进：

1. 先确认教程总目录与章节命名
2. 删除当前初始化示例教程内容
3. 先写第 01、02、03 章，建立主线
4. 同步搭建 `examples/` 目录，保证每章都有可运行 demo
5. 完成 Part I 和 Part II 后，再集中处理 Context / RAG / MCP / Skill
6. 最后统一补导航文案和首页介绍

## 15. 结论

这套教程最关键的不是“覆盖了多少术语”，而是是否让读者在认知上和代码上都完成一次真实的能力升级。

如果整体结构按本文执行，那么教程的核心优势会非常明确：

- 主线清晰，不碎片化
- 从原理到工程实现自然过渡
- 重点放在 Agent 的真实难点，而不是表面热词
- 读者最终能带走一个真正可运行的 code agent

后续正式写作时，建议先从第 01 到第 03 章开始，因为这是整条主线中最容易建立“成就感”的部分，也是后续所有能力的共同基础。
