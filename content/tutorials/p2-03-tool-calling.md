## 本章定位

在最小 Agent Loop 跑通之后，接下来要解决的问题不是“接更多工具”，而是“怎样设计模型更容易正确使用的工具”。

## 本章目标

- 讲清楚工具契约为什么重要
- 让读者学会设计输入 schema、输出结构和错误语义
- 为后面的 ReAct、Plan-and-Execute 和 MCP 打好基础

## 建议内容结构

### 1. 什么叫“适合模型调用”的工具

对比两个版本：

- 一个只接收自由字符串的脆弱工具
- 一个有结构化参数和明确返回值的工具

### 2. 输入要结构化

建议用 TypeScript 类型和 `zod` 一起讲：

- 字段命名
- 必填项
- 枚举值
- 默认值

### 3. 输出要服务下一轮推理

工具返回不应只有“成功/失败”，还应包含：

- 关键结果
- 简要摘要
- 可恢复错误信息

### 4. 错误处理与恢复

建议讲：

- 参数校验错误
- 工具执行错误
- 超时
- 是否可重试

### 5. 搭一个通用工具注册器

把工具定义统一成可注册、可发现、可执行的格式。

## 建议代码产出

- `tool-registry.ts`
- `tool-types.ts`
- `tool-error.ts`
- 至少 3 个示例工具

## 推荐示例工具

- `searchFiles`
- `readFile`
- `fetchUrl`
- `writeTodo`

## 本章写作提醒

- 不要把多个职责塞进一个超级工具
- 要强调“给模型设计接口”和“给人写函数”不是同一件事
- 可以加入错误案例，让读者理解为什么 schema 会直接影响成功率
